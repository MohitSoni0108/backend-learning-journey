Real World Scenario: User Profile API

Suppose you have:

users.json

and 100 users are using your application.

When a user opens their profile:

Browser
   ↓
Node Server
   ↓
Read users.json
   ↓
Send Response
Case 1: Using Synchronous File Reading ❌
const data = fs.readFileSync("users.json", "utf8");

Imagine:

User A Request Arrives
      ↓
Node starts reading file
      ↓
Wait...
Wait...
Wait...
      ↓
File read completed
      ↓
Response sent

During this time:

User B arrives
User C arrives
User D arrives

All are forced to wait.

Because:

Node Call Stack
│
└── readFileSync()
       ↓
   Blocks everything

Node cannot process anything else.

Visual Timeline
Time →

User A  [ Reading File ----------- Done ]

User B                 [ Waiting ]

User C                 [ Waiting ]

User D                 [ Waiting ]

This is called:

Blocking I/O
Case 2: Using Asynchronous File Reading ✅
fs.readFile(
  "users.json",
  "utf8",
  (err, data) => {
    console.log(data);
  }
);

Now:

User A arrives
      ↓
Node delegates file reading to OS
      ↓
Node becomes free immediately

While the OS is reading:

User B arrives
User C arrives
User D arrives

Node can continue handling them.

Visual Timeline
Time →

User A [ OS Reading File ]

User B [ Processing ]

User C [ Processing ]

User D [ Processing ]

Nobody is blocked.

Mental Model

Imagine you're a waiter in a restaurant.

Synchronous Waiter

Customer:

Bring me coffee.

Waiter:

Goes to kitchen
Waits for coffee
Returns

During this time:

Other customers ignored
Asynchronous Waiter

Customer:

Bring me coffee.

Waiter:

Places order in kitchen
Returns immediately
Serves other customers

When coffee is ready:

Kitchen notifies waiter
Waiter serves coffee

This is exactly how:

fs.readFile()

works.

Why This Matters for Backend

Imagine:

Amazon
Instagram
Netflix

Millions of users.

If every file operation was synchronous:

User 1 → Read File
Wait

User 2 → Wait

User 3 → Wait

User 4 → Wait

Server becomes slow.

With asynchronous operations:

User 1 → Reading

User 2 → Reading

User 3 → Reading

User 4 → Reading

User 5 → Reading

All requests can be handled efficiently.

When is Sync Actually Used?

Synchronous methods aren't evil.

They're useful for:

Startup Configuration
const config = fs.readFileSync(
  "config.json",
  "utf8"
);

Server starts once.

Waiting is acceptable.

Small Scripts
node script.js

For CLI tools:

read file
process file
exit

Sync code is often simpler.

Rule of Thumb
Building Server/API
        ↓
Use Async

Building Small Script
        ↓
Sync is Fine
The Deep Connection

Remember your Node Runtime mental model:

Node Runtime
│
├── Call Stack
├── Event Loop
├── Node APIs
└── OS Threads

When you do:

fs.readFile()

Node says:

"OS, please read this file for me."

and immediately continues.

When the file is ready:

OS
 ↓
Callback Queue
 ↓
Event Loop
 ↓
Call Stack

This is the same mental model as:

setTimeout()
fetch()
fs.readFile()
database queries

All of them are asynchronous I/O operations.

One-Line Memory Trick
readFileSync()
=
Node waits for the file

readFile()
=
Node asks the OS to read the file and keeps working

This is one of the biggest reasons Node can handle thousands of concurrent requests despite having a single JavaScript thread. 🚀
